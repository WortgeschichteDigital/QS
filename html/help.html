Aufgabe der App

QS ist ein Tool zur Qualitäts-Sicherung für Wortgeschichte digital (WGd). Es hat zwei Kernfunktionen: Zum einen scannt es die von WGd erstellten XML-Dateien auf Fehler und Verlinkungsvorschläge, wobei es hier nur Aufgaben übernimmt, die durch das von WGd verwendete RelaxNG-Schema nicht abzudecken sind. Zum anderen können mit ihr die WGd-Verweiscluster analysiert und v. a. auch moduliert werden.

Über allem steht dabei das Ziel, dass die Mitarbeiter*innen des Projekts ihre XML-Dateien möglichst eigenständig und ohne Reibungsverluste für die Publikation vorbereiten können. Da mithilfe von QS die Datendatei Artikel.json von jedermann mit nur ein paar Klicks erstellt werden kann, stellt QS zugleich sicher, dass die Publikation neuer Wortgeschichten auch ohne technische Spezialist*innen möglich bleibt. Denn ohne die Neuberechnung dieser Datendatei ist eine Publikation von Wortgeschichten gleichsam undenkbar, da sie auf der Webseite zahlreiche Funktionen füttert:

  • die Suchfilter der Artikelübersicht,
  • die Liste der neuen Artikel in der Artikelübersicht,
  • die Cluster (in der Artikelübersicht und in den Wortgeschichten),
  • den Zeitstrahl,
  • die Suchleiste in den Wortgeschichten und
  • die automatischen Verknüpfungen mit WGd-Artikeln im Belegnavigator.

Darüber hinaus kann QS die Artikelübersicht und den Index im terminologischen Kerninventar auffrischen. Außerdem ist die App in der Lage, die Wortverlaufskurven an das ZDL-Farbschema anzupassen. Diese Funktionen werden im Abschnitt Publikation erläutert.

Für Entwickler*innen: Die Publikationsfunktionen lassen sich alle via CLI ausführen.

Überblick

Die App hat vier Ansichten, die über den Kopf des Hauptfensters angesteuert werden können: XML, Hinweise, Clusterei und Suche. Die Daten für die Operationen und Anzeigen dieser Ansichten speisen sich aus einem lokalen Klon des Wortgeschichten-Repositorys, der zunächst eingerichtet werden muss. Ohne Repository im Hintergrund ist die App nicht lauffähig. Darum werden beim ersten Start auch der Ort und der Benutzername für dieses Repository erfragt.

Alle Git-Befehle, die für die Benutzer*innen relevant sind, können ausnahmslos über die App ausgeführt werden. Wie die Ansichten können sie über den Kopf des Hauptfensters aufgerufen werden.

Um QS zu nutzen, muss niemand in der Lage sein, Commits mit Git zu erstellen. Man kann einfach neue XML-Dateien in den Ordner ignore legen, auf Update drücken (Tastaturkürzel F5) und die App frischt ihre Daten unter Berücksichtung der geänderten oder ergänzten Dateien auf. Die Analyse der XML-Dateien, auf deren Ergebnis alle vier Ansichten zurückgreifen, ist rechenintensiv, weswegen sie in einem Cache zwischengespeichert wird, auf den auch nach dem Neustart des Programms zurückgegriffen wird. XML-Dateien werden nur dann neu analysiert, wenn die App feststellt, dass sie verändert oder neu hinzugefügt wurden. Und nur dann, wenn eine XML-Datei neu analysiert wurde, werden auch die Hinweise zu den Dateien neu berechnet. Eine Ausnahme stellt hier die Installation einer neuen Programmversion dar. In diesem Fall werden der Cache zurückgesetzt, die XML-Dateien komplett neu eingelesen und alle Daten neu berechnet.

Filterleiste

Die Filterleiste kann über den Button im Fensterkopf oder auch mit dem Tastaturkürzel Strg + Umsch + F geöffnet bzw. geschlossen werden. Der Inhalt der Filterleiste passt sich an die aktuelle Ansicht an. Filter, die für eine Ansicht nicht relevant sind, werden ausgeblendet. Die Auswahl der Filter wird beim Beenden der App gespeichert und beim Start wiederhergestellt.

Die Funktion zurücksetzen im Kopf der Filterleiste setzt zwar alle Filter auf ihren Standardwert zurück. Es werden aber nur diejenigen Filter geändert, die in der aktuellen Ansicht sichtbar sind.

Die Filter im Einzelnen:

Autor*innen
Es werden nur XML-Dateien der ausgewählten Autor*in berücksichtigt. Für die Clusterei bedeutet dies, dass in den Sektionen Vergleichen und Überprüfen nur Cluster sichtbar sind, deren Zentrum mindestens ein Lemma hat, das von der ausgewählten Autor*in behandelt wurde. Die Berechnung der Vorschau hingegen berücksichtigt immer alle Lemmata.

Themenfelder
Die Analyse bzw. Anzeige der XML-Dateien wird in allen Ansichten auf das ausgewählte Themenfeld begrenzt. Um die Sektionen Vergleichen und Überprüfen in der Clusterei nutzen zu können, müssen Sie zwingend ein Themenfeld auswählen.

Dateistatus
Liegt eine XML-Datei im Ordner ignore, hat sie nur dann den Status geändert, wenn sie von einer Datei gleichen Namens im Ordner articles inhaltlich abweicht. Neu sind Dateien, die im Ordner ignore liegen, aber nicht im Ordner articles oder sogenannte untracked files im Ordner articles. Dies sind Dateien, die noch nie committet wurden, deren Existenz von Git wahrgenommen, die ansonsten aber ignoriert werden.

Diese Filterkategorie ist nur in der Ansicht Hinweise sichtbar. Die Filter können dafür genutzt werden, entsprechend markierte Hinweise ein- oder auszublenden.

Allgemeines

Git ist ein verteiltes Versionskontrollsystem, das 2005 von Linus Torvalds, dem Erfinder von Linux, entwickelt wurde. Geschrieben wurde es für die Quellcodeverwaltung des Linux-Kernels. Es soll sukzessive Änderungen an Dateien nachvollziehbar machen und es jederzeit ermöglichen, ältere Versionen dieser Dateien wiederherzustellen.

Git ist insofern ein verteiltes Versionskontrollsystem, als es im Gegensatz zu einem zentralen System keine Instanz gibt, von der man abhängig oder an die man gebunden ist. Es ist vielmehr so, dass jede Benutzer*in eines Projekts einen lokalen Klon des kompletten Quelltexts dieses Projekts inklusive seiner Geschichte auf dem eigenen Rechner speichert. Die einzelnen, unabhängig voneinander arbeitenden Instanzen von Git können sich dann über einen Austauschknoten miteinander synchronisieren. Im Falle von WGd ist dieser Knoten ein Klon des Repositorys, der bei ZDL Git liegt. Der Austausch findet über sogenannte Pushes und Pulls statt. Für die WGd-Miterarbeiter*innen ist in der Regel nur der Befehl pull interessant, für den es eine Funktion im Kopf des Hauptfensters von QS gibt.

Git speichert die Versionsgeschichte der verwalteten Dateien in einem sogenannten Repository. Ein Repository ist nichts anderes als eine Datenbank, in der die Zustände der verwalteten Dateien aufbewahrt und aus der die unterschiedlichen Zustände der Dateien bei Bedarf wieder abgerufen werden können. Für Benutzer*innen gibt es eine einfache, jedermann zugänglich Bedienoberfläche: Bei ihr handelt es sich um nichts anderes als einen ganz gewöhnlichen Ordner mit Dateien, der an einem selbstgewählten Ort auf dem eigenen Betriebssystem liegt. Wenn man tiefer in die Materie einsteigt, muss man zwar Textbefehle tippen können, das ist für die Benutzung von QS aber ganz und gar unnötig.

Wie eigentlich jedes Versionskontrollsystem kennt auch Git sogenannte Branches. Branches sind Abzweigungen in der Codebasis. Mithilfe von Branches kann man mehrere Features einer Software parallel entwickeln. Für WGd sind Branches deswegen wichtig, weil wir auf diese Weise das Repository in publizierte und noch nicht publizierte Artikel auftrennen können.

Für die Mitarbeiter*innen von WGd sind darum zwei Branches relevant: master und preprint. Die XML-Dateien, die unter master liegen, sind publiziert. Die Inhalte dieser Dateien werden in den Wortgeschichten auf zdl.org angezeigt. Der Branch preprint hingegen ist für die Vorbereitung von Publikationen gedacht. Die XML-Dateien neuer Artikel werden zunächste nach preprint geladen und erst später, wenn ein Imprimatur erteilt wurde, nach master gemergt, also in den anderen Branch übertragen. Steht keine Publikation an, sind die Inhalte von master und preprint vollkommen identisch. (Dafür sorgen die Mitarbeiter*innen, die sich genauer mit Git auskennen.)

Es ist nie so, dass in einem Branch nur die Dateien der einen und im anderen nur die Dateien der anderen Kategorie liegen. Die Dateien der Branches master und preprint teilen immer eine gemeinsame Basis. Und diese Basis ist der Punkt, an dem die Abzweigung im Code vorgenommen wurde. Darum finden Sie in preprint auch bereits publizierte XML-Dateien. Aber Vorsicht! Diese können unter Umständen inhaltlich von denen in master abweichen, was immer dann passiert, wenn eine Mitarbeiterin nach dem Erstellen der Abzweigung an den Dateien in master Änderungen vorgenommen hat. Auf denselben Stand gebracht werden die beiden Branches erst dann, wenn preprint nach master gemergt wird, wenn die beiden Branches also miteinander verschmolzen werden. Dies passiert – normalerweise – dann, wenn Artikel publiziert werden.

Mit dem Befehl checkout können Sie jederzeit den Branch wechseln. Git sorgt dann automatisch dafür, dass die Dateien im Ordner Ihres Repositorys an die im jeweiligen Branch zu findende Version angepassst werden. Dies kann unter Umständen auch bedeuten, dass Dateien neu erscheinen oder verschwinden.

Git installieren

Windows: Installationsmedien kann man von der Git-Webseite herunterladen.

macOS: Git ist wahrscheinlich schon installiert. Ob dem wirklich der Fall ist, kann man überprüfen, indem man das Programm Terminal öffnet und diesen Befehl ausführt: $ git --version. Wird eine Versionsnummer angezeigt, ist git installiert. Ist Git nicht installiert, muss man die Xcode Command Line Tools installieren. Dazu müssen Sie in Terminal diesen Code ausführen: $ xcode-select --install.

Linux: Git ist wahrscheinlich schon installiert. Wenn nicht führen Sie $ sudo apt-get git aus (oder den für Ihr System passenden Code). Auch hier kann natürlich mit $ git --version überprüft werden, ob Git bereits installiert ist.

Respository einrichten

  1. Sie benötigen einen Zugang zu ZDL Git.
  2. Git Bash (unter Windows) oder Terminal (unter macOS) an der Stelle öffnen, an der das Repository erstellt werden soll.

  3. Wortgeschichten-Repository klonen:

    $ git clone https://git.zdl.org/zdl/wortgeschichten.git

  4. In den Repository-Ordner wechseln:

    $ cd wortgeschichten

  5. Git konfigurieren. Dafür die folgenden Befehle nacheinander ausführen; den gefetteten Text entsprechend anpassen:

    $ git config user.name "Petra Pan"
$ git config user.email peter.pan@adwgoe.de
$ git config pull.rebase true
$ git config core.editor nano

  6. Nur unter macOS und Linux:

    $ git config credential.helper cache

    Wenn Sie XML-Dateien in Ihrem Repository committen, die auf einem Windows-System bearbeitet wurden:

    $ echo '*.xml text eol=lf' > .gitattributes
$ echo "/.gitattributes" >> .git/info/exclude

  7. Nur unter Windows:

    $ git config core.autocrlf true

  8. Preprint-Branch einrichten:

    $ git switch --track origin/preprint

    Sollte switch in Ihrer Git-Version nicht zur Verfügung stehen, geht auch dieser Befehl:

    $ git checkout -b preprint --track origin/preprint

  9. Ordner ignore erzeugen:

    $ mkdir ignore

  10. Ordner ignore vor Git verstecken:

    Hierzu müssen Sie folgende Konfigurationsdatei öffnen: .git/info/exclude.

    Am Ende der Datei, hinter der letzten Zeile, die mit # beginnt, ergänzen Sie diese Zeile:

    /ignore

  11. Status anzeigen lassen:

    $ git status

    Der Befehl sollte den folgenden Text auswerfen. Ist dem der Fall, hat alles geklappt.

    On branch preprint
Your branch is up to date with 'origin/preprint'.

nothing to commit, working tree clean

Dateien und Ordner

Das Repository befindet sich nach der Einrichtung im Ordner wortgeschichten. Sie können den Ordner jederzeit umbenennen oder verschieben. Dafür sind keine Git-Befehle nötig. Die Konfiguration von Git wird in dem versteckten Ordner .git gespeichert. Umbenennen und/oder Verschieben des übergeordneten Ordners tangiert die Konfiguration nicht.

Das Repository besteht aus folgenden Dateien und Ordnern:

  • .git versteckter Ordner mit der Git-Konfiguration und dem Repository im engeren Sinne
  • articles/ Ordner mit XML-Dateien
  • resources/ Datendateien und Abbildungen
    • resources/images/ Abbildungen
    • resources/images/qrc/ QR-Codes für den Footer der Wortgeschichten
    • resources/images/thumbs/ Vorschaubilder für JPEG-Abbildungen
    • resources/Artikel.json Datendatei mit Artikeldaten
    • resources/Quiz.json Datendatei fürs Wort-Quiz
    • resources/Register.json Datendatei mit Registerwörtern
    • resources/Terminologie.json Datendatei mit den Texten des terminologischen Kerninventars
  • share Ordner mit Schema, Musterdateien und Literaturliste
    • share/rnc/ das RelaxNG-Schema für die XML-Dateien
    • share/tei/ Musterdateien im WGd-XML-Format und in TEI
    • share/Literaturliste.xml XML-Dateien mit allen Titeln aus der Literaturdatenbank von Zettel’s Traum
  • README.md Lies-mich-Datei mit Infos zum Repository

Der Ordner ignore gehört im eigentlichen Sinne nicht zum Repository, weil seine Inhalte von Git nicht beobachtet und somit auch nicht verwaltet werden.

Die Ordner articles, resources, share und die Datei README.md hingegen werden von Git beobachtet. Diese Ordner und die in ihnen enthaltenen Dateien können mit dem Befehl restore zurückgesetzt werden. Denn Git kümmert sich um die bereits bekannten Dateien in diesen Ordnern. Außerdem werden die Ordner von Git, wie gesagt, beobachtet: Legt man neue Dateien in einen der beobachteten Ordner oder direkt in den Ordner wortgeschichten, wird Git dies registrieren, sich um diese Dateien erst einmal aber nicht weiter kümmern. Wenn Git eine Datei aber bereits bekannt ist, macht es einen durch den Befehl status gegebenenfalls darauf aufmerksam, dass sie geändert wurde.

Wurde bspw. die Datei Abgehaengter.xml geändert, erscheint dieser Status-Text:

On branch master
Your branch is up to date with 'origin/master'.

Changes not staged for commit:
  (use "git add <file>..." to update what will be committed)
  (use "git restore <file>..." to discard changes in working directory)
        modified:   articles/Abgehaengter.xml

no changes added to commit (use "git add" and/or "git commit -a")

Wenn Sie in einem der von Git beobachteten Ordner eine neue, Git noch unbekannte Datei ablegen, handelt es sich dabei um eine sogenannte untracked file. Git registriert zwar, dass es eine neue Datei gibt, macht mit dieser aber erst einmal nichts. Darum werden untracked files durch den Befehl restore auch nicht zurückgesetzt – Git sieht diese Dateien, fühlt sich für sie aber (noch) nicht zuständig.

Im folgenden Fall wurde im Ordner articles die neue Datei Handtasche.xml erstellt:

On branch master
Your branch is up to date with 'origin/master'.

Untracked files:
  (use "git add <file>..." to include in what will be committed)
        articles/Handtasche.xml

nothing added to commit but untracked files present (use "git add" to track)

Was heißt das alles für Sie in der Praxis? Ganz einfach: Arbeiten Sie niemals in den von Git beobachteten Ordnern! Wenn Sie eine XML-Datei bearbeiten möchten, kopieren Sie diese und legen Sie die Kopie im Ordner ignore ab. In der XML-Ansicht gibt es hierfür mit dem Icon eine Möglichkeit, dies mit einem Klick zu tun. Denn der Ordner ignore und seine Inhalte werden von Git nicht beobachtet. Da er aber im Repository relativ dieselbe Position wie der Ordner articles hat, erreichen die dort befindlichen XML-Dateien trotzdem das Schema, das im Ordner share/rnc liegt. So kann das XML dieser Dateien beim Bearbeiten automatisch validiert werden. Für den Ordner ignore gilt übrigens dasselbe wie für die untracked files: Der Befehl restore entfaltet in ihm keine Wirkung.

Befehle

Für die Mitarbeiter*innen von WGd sind im Arbeitsalltag nur vier Git-Befehle relevant:

  • checkout (Wechseln des Branches),
  • status (Anzeige des Zustands, in dem sich das Repository befindet),
  • pull (neuste Dateiversionen herunterladen) und
  • restore (Repository zurücksetzen).

Die Befehle lassen sich alle über die klicksensitiven Texte im Kopf des Hauptfensters bedienen. Nach dem Absetzen der Befehle checkout, pull und restore stößt die App automatisch ein Update an, wodurch die Daten, mit denen QS arbeitet, auf dem neusten Stand gebracht werden.

Befehl checkout
Der Befehl dient zum Wechseln des Branches. In der Anzeige steht in diesem Fall nicht der Name des Befehls, sondern der Name des Branches, in dem sich das Repository gerade befindet: master oder preprint. Durch Klick auf den Text wird in den anderen Branch, also den anderen Zweig im Code gewechselt.

Welchen Branch soll ich nutzen? Faustregel: Für das Bearbeiten neuer, noch nicht publizierter XML-Dateien, sollten Sie immer preprint nutzen. Für das Bearbeiten älterer, bereits publizierter XML-Dateien hingegen master.

Befehl status
Der Befehl öffnet ein Meldungsfenster, das den aktuellen Status des Repositorys anzeigt. Mitarbeiter*innen brauchen diesen Befehl normalerweise nicht. Benötigt man aber Hilfe, weil irgendetwas mit dem Repository nicht zu stimmen scheint, ist es gut, den im Meldungsfenster angezeigten Text zu kopieren und an eine kompetente Mitarbeiter*in zu schicken.

Endet die Statusanzeige mit der Zeile nothing to commit, working tree clean, ist alles in Ordnung.

Befehl pull
Mit diesem Befehl kann man sich die neusten Versionen der XML-Dateien von WGd herunterladen. Genauer gesagt werden beim Pullen vom zentralen Austauschknoten sogenannte Commits heruntergeladen. Diese Commits sind kleine Päckchen, die Änderungen am Quelltext des Projekts speichern. Konkret: In ihnen stehen Änderungen, die irgendjemand an einer XML-Datei vorgenommen hat, oder sogar komplette, neu hinzugefügte Dateien. Um den eigenen Klon des Repositorys, sprich: den Ordner mit XML-Dateien auf dem eigenen Computer immer auf dem neusten Stand zu halten, sollten Sie öfter mal einen Pull ausführen.

Das RelaxNG-Schema, das die XML-Dateien auf Validität überprüft, liegt auch im Repository (im Ordner share/rnc). Beim Pullen werden nicht nur die XML-Dateien, sondern auch das Schema automatisch auf den neusten Stand gebracht. Vor dem Bearbeiten von XML-Dateien sollten Sie darum immer pullen, weil Sie sonst Gefahr laufen, nicht-valide Dateien zu produzieren.

Führen Sie den Pull-Befehl aus, erscheint ein Dialog, der Ihr Passwort für das ZDL Git erfragt. Das Passwort wird zwischengespeichert, sodass Sie es nur einmal pro Programmsession eingeben müssen. Beim Schließen der App wird es allerdings gelöscht, aus Sicherheitsgründen.

Befehl restore
Mit diesem Befehl können Sie das Repository zurücksetzen. Zurücksetzen bedeutet, dass alle Dateien auf den Stand des jüngsten Commits gebracht werden. Das heißt nichts anderes als dies: Haben Sie Änderungen an XML-Dateien vorgenommen und setzen sie das Repository zurück, gehen die Änderungen unwiederbringlich verloren. Dabei werden Dateien im Ordner ignore allerdings niemals angetastet. Diese sind vor den Konsequenzen von restore in jedem Fall geschützt.

Warum gibt es diese Funktion? Wenn Sie eine XML-Datei bearbeiten möchten, sieht der Workflow normalerweise so aus: Sie kopieren die Datei aus dem Ordner articles und legen eine Kopie im Ordner ignore ab. Diese Kopie im Ordner ignore öffnen Sie dann zum Bearbeiten. Sind Sie mit der Arbeit fertig, reichen Sie die Kopie an eine kompetente Mitarbeiter*in weiter, die die Änderungen committet und pusht, sodass sie allen Mitarbeiter*innen nach einem Pull zur Verfügung stehen.

Wenn Sie die XML-Datei aber aus Versehen nicht etwa kopieren, sondern ausschneiden oder verschieben, entfernen Sie dadurch die Originaldatei aus dem Ordner articles. Da articles ein von Git beobachteter Ordner ist, nimmt Git das Fehlen der Datei als Änderung am Repository wahr und Sie bekommen früher oder später Probleme mit dem Pullen, weil sich der Status Ihres Repositorys geändert hat. Außerdem wird sich QS weigern, den Branch zu wechseln, weil der working tree nicht mehr clean ist. Kopieren Sie die nunmehr fehlende Datei einfach wieder zurück, ist diese von Ihnen vielleicht schon geändert worden, was die Lage nicht besser macht: Git wird sie immer noch als verändert registrieren und der Pull könnte zum Problem werden. Die einfachste Lösung ist in so einem Fall ein Restore. Durch den Befehl wird die fehlende Datei von Git im Ordner articles neu erzeugt, und zwar mit denselben Inhalten, die sie zuvor auch hatte. Aber Vorsicht! Haben Sie eine Datei vor dem Bearbeiten nicht verschoben, sondern direkt im Ordner articles bearbeitet, gehen die Änderungen an dieser Datei durch Klick auf restore unwiederbringlich verloren!

Weil restore gravierende Folgen haben kann, fragt QS vor dem Ausführen des Befehls immer noch einmal nach, ob Sie ihn wirklich aufrufen möchten.

Aufbau der Ansicht

Bei dieser Ansicht handelt es sich um nichts anderes als eine Liste der im Repository befindlichen XML-Dateien. Eingelesen werden sowohl die Git bereits bekannten Dateien aus dem Ordner articles als auch die XML-Dateien aus dem Ordner ignore. Neue XML-Dateien müssen also nicht erst durch einen Commit in das Repository eingespeist werden. Es genügt, wenn Sie diese im Ordner ignore ablegen. QS kann mit diesen Dateien all das machen, was es auch mit Dateien aus articles machen kann.

Sie können die Liste nach drei Kategorein filtern: Autor*in, Themenfeld und Dateistatus. Zusätzlich können sie in der Kopfleiste noch den Pfad-Filter nutzen.

Für jede Datei stehen Ihnen mehrere Operationen zur Verfügung:

  • Ein Klick auf den Dateinamen öffnet die Datei im Vorschaufenster;
  • mit dem Icon können Sie eine Kopie der Datei in den Ordner ignore legen;
  • mit dem Icon können Sie den XML-Quelltext der Datei in die Zwischenablage kopieren;
  • mit dem Icon können Sie sich das Kurz-Gefasst der Datei anzeigen lassen, und zwar in der Form, in der es in den Suchergebnissen von zdl.org erscheint;
  • mit dem Icon können Sie die Datei im Standard-XML-Editor Ihres Betriebssystems öffnen.

Vorschaufenster

Das Vorschaufenster ruft die XML-Vorschaufunktion von zdl.org auf, die Sie auch im Browser nutzen können. Was im Browser viele einzelne Handgriffe erfordert, geht in QS freilich automatisch: Die App holt sich den XML-Code der gewünschen Datei von der Festplatte, und schickt ihn direkt an die XML-Vorschau auf zdl.org, wo er dann nach HTML transformiert wird. Sie können im Vorschaufenster beliebig navigieren, also auch im Artikel verlinkte Webseiten außerhalb von zdl.org aufrufen.

Es können beliebig viele Vorschaufenster geöffnet werden. Klicken Sie allerdings auf eine Datei, für die bereits ein Vorschaufenster existiert, wird kein neues Fenster erzeugt, sondern das bestehende genutzt.

Nehmen Sie Änderungen an der XML-Datei vor, müssen Sie im Vorschaufenster nur noch das Update-Icon klicken und Sie sehen den Artikel in just der Fassung, an der Sie gerade arbeiten. Aufbau der Hinweise

Die Hinweise sind immer gleich aufgebaut:

  • Die Zeilenangabe am linken Rand gibt an, wo in der XML-Datei sich das Element, auf das sich der Hinweis bezieht, befindet. Bei Elementen, die sich über mehr als eine Zeile erstrecken, wird immer die erste Zeile genannt.
  • Jeder Hinweis beginnt mit einer Nennung des Hinweistyps. Die Hinweistypen sind mit ihrer Beschreibung im Handbuch verlinkt.
  • Der Textblock mit vorangestelltem beschreibt einen Fehler.
  • Der Textblock mit vorangestelltem gibt einen Korrekturhinweis. Wird korrekter XML-Code vorgeschlagen, kann man diesen durch einfachen Klick automatisch kopieren.
  • Einige Hinweise (z. B. XML-Kommentar) haben keines dieser Icons. Sie sind rein informationeller Natur, sollen ein Phänomen also weder als richtig noch als falsch markieren.
  • Das Icon gibt grob den Ort in der XML-Datei an, auf den sich der Hinweis bezieht. So kann man z. B. sofort sehen, ob sich ein Verweis im Verweisblock oder Text der Wortgeschichte befindet.

Auch Hinweise, in denen ein Phänomen als fehlerhaft markiert ist, sind nicht notwendigerweise Fehler. Dies ist häufig eine Frage der Auslegung. Auch bei diesen Hinweisen handelt sich zunächst einmal nur um Vorschläge, die bedacht, oder Informationen, die zur Kenntnis genommen werden sollten.

Funktion der Icons unterhalb der Zeilenangabe:

Durch Klick auf dieses Icon kann man den Hinweis markieren. Die Markierung wird gespeichert und beim Neustart des Programms wiederhergestellt. Es gibt zwei verschiedene Markierungen: und . Haben Sie Hinweise markiert, können Sie diese mithilfe des entsprechenden Filters wiederfinden. Die Funktion dient primär dazu, Hinweise als geprüft zu taggen, um sie nicht immer wieder aufs Neue überprüfen zu müssen. In den Einstellungen gibt es eine Funktion, mit der Sie alle Markierungen löschen können.

Die einzelnen Hinweise haben keine ID, aber eine Identifizierung. Diese ist zwar in vielen Fällen eindeutig, aber eben nicht immer. Darum kann es sein, das Sie einen der Hinweise markieren, wodurch automatisch andere Hinweise mitmarkiert werden. Dies kann nur passieren, wenn inhaltlich identische Hinweise in ein und derselben XML-Datei zu finden sind. In solchen Fällen kann allein die Zeilennummer der Hinweise abweichend sein. Da die Zeilennummer aber dynamisch ist (weil davor jederzeit Zeilen hinzugefügt oder entfernt werden könnten), kann diese nicht für die Berechnung der Indentifizierung herangezogen werden.

Ein Klick auf das Icon öffnet ein Popup-Fenster, in dem Sie den Hinweis im Kontext des Originalquellcodes einsehen können, ohne dafür eigens die XML-Datei öffnen zu müssen. Im Popup-Fenster wird der XML-Quelltext zeilenweise dargestellt. Am linken Rand finden Sie die Zeilenzählung. Die Zeile, in der das Element zu finden ist, auf das sich der Hinweis bezieht, ist blau dargestellt. In vielen Fällen wird sogar die genaue Stelle im Quelltext markiert, um die es geht. (Es kann passieren, dass auch andere Stellen mit einem ähnlichen Phänomen hervorgehoben werden.) Der dargestellte Quelltext kann mal mehr, mal weniger umfangreich sein. Findet sich ein Phänomen im Autor*innentext der Wortgeschichte, wird immer ein kompletter Codeblock gedruckt.

Die Hinweise beziehen sich – natürlich – immer auf eine bestimmte XML-Datei. Der Name der XML-Datei, auf die sich ein Hinweis bezieht, bleibt am oberen Rand der Ansicht sichtbar. Neben dem Dateinamen finden Sie drei Icons:

Das Icon schickt den XML-Code der Datei an ein neues Programmfenster. In diesem wird die Vorschau auf zdl.org aufgerufen (mehr zum Vorschaufenster).

Das Icon öffnet ein Popup-Fenster, in dem die Lemmata aufgelistet werden, die in der XML-Datei beschrieben werden. Enthält die Datei einen Wortfeldartikel, werden die Wörter des Wortfeldes aufgelistet.

Das Icon öffnet die Datei im Standard-XML-Editor des Betriebsystems.

Duplikat entfernen

Skopus:<Verweise>

Tag:<Verweis>

In den Wortinformationen des Artikels befindet sich bereits ein identischer Verweis, der auf dasselbe Lemma derselben Wortgeschichte verweist. Das Duplikat sollte entfernt werden.

Artikel-Link korrigieren

Skopus:<Verweise>, <Wortgeschichte>, <Wortgeschichte_kompakt>

Tag:<Verweis>

Hinter diesem Hinweis verbirgt sich eine Vielzahl verschiedener Fehler:

Sprungziel nicht gefunden
In der Wortgeschichte wurde kein Attribut @xml:id gefunden, das zum Sprungziel, welches im Tag <Verweisziel> angegeben wurde, passt. Das Sprungziel wird hinter dem Hash # notiert. Es muss sich im Zielartikel zwingend innerhalb von <Wortgeschichte> befinden. Das Problem liegt also hier:

<Verweis>
  <Verweistext>Elite</Verweistext>
  <Verweisziel>Elite#existiert_nicht</Verweisziel>
</Verweis>

Verweistext passt nicht zum Verweisziel
Wird auf einen Artikel verwiesen, der mehr als ein Hauptlemma hat, sollte der Inhalt von <Verweistext> mit dem von <Verweisziel> kongruent sein. Es sollte also auf das Hauptlemma verwiesen werden, das mit dem Verweis gemeint ist. Die korrekte Auszeichnung ist für die Berechnung der Verweiscluster von großer Bedeutung. (Dieser Hinweis produziert häufig falschpositive Treffer, die natürlich ignoriert werden sollten.)

<Verweis>
  <Verweistext>Kleinbürger</Verweistext>
  <Verweisziel>Großbürger</Verweisziel>
</Verweis>
<Verweis>
  <Verweistext>Kleinbürger</Verweistext>
  <Verweisziel>Kleinbürger</Verweisziel>
</Verweis>

Wortfeldartikel falsch verlinkt
Normalerweise wird auf Artikel verwiesen, indem man in <Verweisziel> die erste Schreibung des ersten Hauptlemmas angibt. Bei Wortfeldartikeln muss dieser Schreibung allerdings Wortfeld- vorangestellt werden.

<Verweis>
  <Verweistext>sozialräumliche Segregation</Verweistext>
  <Verweisziel>sozialräumliche Segregation</Verweisziel>
</Verweis>
<Verweis>
  <Verweistext>sozialräumliche Segregation</Verweistext>
  <Verweisziel>Wortfeld-sozialräumliche Segregation</Verweisziel>
</Verweis>

Lemma nicht gefunden
Das Lemma, auf das in <Verweisziel> verwiesen wird, wurde nicht gefunden. Ist geplant, eine Wortgeschichte zum Lemma zu publizieren, sollte der fehlerhafte Verweis nicht entfernt, sondern auskommentiert werden. Auf diese Weise kann er, sobald das Lemma online steht, leicht wieder scharfgeschaltet werden.

<!--<Verweis>
  <Verweistext>--><erwaehntes_Zeichen>Elite</erwaehntes_Zeichen><!--</Verweistext>
  <Verweisziel>Elite</Verweisziel>
</Verweis>-->

Verweistext nicht angegeben
Enthält <Verweisziel> eine Sprungposition, muss zwingend ein <Verweistext> angegeben werden. Ansonsten wird der Text von <Verweisziel> – Hash inklusive – auf der Webseite gedruckt.

<Verweis>
  <Verweistext/>
  <Verweisziel>Slum#Verslumung_Zunehmende_Verwahrlosung_von_Städten</Verweisziel>
</Verweis>
<Verweis>
  <Verweistext>Verslumung</Verweistext>
  <Verweisziel>Slum#Verslumung_Zunehmende_Verwahrlosung_von_Städten</Verweisziel>
</Verweis>

Nebenlemma anstelle von Hauptlemma
In <Verweisziel> steht ein Nebenlemma, was nicht korrekt ist. Denn Verweise auf Nebenlemmata werden korrektweise so konstruiert: 1. Schreibung des 1. Hauptlemmas + # + Sprungziel des Nebenlemmas. (Das Sprungziel kann in der XML-Datei aus dem Attribut @Ziel der <Textreferenz>, die zum entsprechenden Nebenlemma gehört, ausgelesen werden.) Arbeitspraktisch ist es ein gutes Vorgehen, während des Artikelschreibens erst einmal nur die Nebenlemmata anzugeben. Vor der Publikation der Wortgeschichte muss dies aber unbedingt korrigiert werden. Im Hinweis wird das korrekte <Verweisziel> angegeben, das mit einem Klick kopiert werden kann.

<Verweis>
  <Verweistext><erwaehntes_Zeichen>Normalverbraucher</erwaehntes_Zeichen></Verweistext>
  <Verweisziel>Normalverbraucher</Verweisziel>
</Verweis>
<Verweis>
  <Verweistext><erwaehntes_Zeichen>Normalverbraucher</erwaehntes_Zeichen></Verweistext>
  <Verweisziel>Otto Normalverbraucher#Der_Normalverbraucher_in_der_Kriegs-_und_Nachkriegswirtschaft</Verweisziel>
</Verweis>

Textreferenz korrigieren

Tag:<Textreferenz>

Das im Attribut @Ziel der <Textreferenz> angegebene Sprungziel wurde nicht gefunden. Das Sprungziel muss sich zwingend innerhalb von <Wortgeschichte> befinden.

Tag:<Verweis_extern>

Die Struktur von <Verweis_extern> ist formal inkorrekt. Dies kann mehrere Gründe haben:

Fehlender Verweistext
Wenn kein <Verweistext> angegeben wurde, ist der Verweis auf der Webseite nicht zu sehen.

Fehlende URL
Ohne <URL> führt der Verweis ins Leere oder, genauer gesagt, zum Artikel selbst.

Externe Verweise dürfen kein Attribut @Typ="Cluster" haben. Dieses Attribut ist nur in Bezug auf interne Verweise sinnvoll. Hat ein externer Verweis dieses Attribut, wird er auf der Webseite ausgeblendet, müsste also gar nicht erst gesetzt werden. Für die Berechnung der Verweiscluster spielen externe Verweise nebenbei gesagt keine Rolle.

Objektsprache mit Artikel verlinken

Tag:<erwaehntes_Zeichen>

Für das Wort in <erwaehntes_Zeichen> gibt es eine eigene Wortgeschichte. Es könnte darum sinnvoll sein, sie an dieser Stelle zu verlinken.

Wenn es bereits einen Verweis auf die entsprechende Wortgeschichte in <Wortgeschichte> oder <Wortgeschichte_kompakt> gibt, wird dies am Ende des Hinweises explizit angegeben. Dort steht dann, wie oft im Autor*innentext des Artikels bereits auf dieses Lemma verwiesen wurde. Stichwort falsch ausgezeichnet

Dieser Hinweis erscheint, wenn <erwaehntes_Zeichen> ein Lemma (Haupt- oder Nebenlemma) des aktuellen Artikels auszeichnet. In diesem Fall sollte für die Auszeichnung unbedingt <Stichwort> genutzt werden.

Das Attribut @Sprache ist an der gemeldeten Stelle überflüssig, weil sein Wert impliziert ist. Diasystematik ergänzen

Für diesen Hinweis werden alle diasystematischen Angaben eingelesen, die im RelaxNG-Schema zu finden sind. Wird im Text einer <Lesart> eine Zeichenfolge gefunden, auf die eine der möglichen diasystematischen Angaben passt, wird ein Hinweis diesen Typs ausgegeben.

Sie können den Vorschlag in zweierlei Weise umsetzen. Zum einen können Sie die Angabe im dafür vorgesehen Tag am Anfang einer Lesart einfügen. Auf der Webseite wird die diasystematische Angabe in diesem Fall nachgestellt:

<Lesart n="1" xml:id="l1-1">
  <Diasystematik>
    <Register>abwertend</Register>
  </Diasystematik>
  <Textreferenz/>

Tagging: Semantik

Semantik ergänzen

Skopus:<Verweise>, <Wortgeschichte>, <Wortgeschichte_kompakt>

Tag:<Verweis>

Verweise können mit lexikalischen Semantiken wie Gegensatz oder Synonym angereichert werden, indem man das Attribut @Typ ergänzt. Die Semantik steht dabei immer in Relation zu einem der Lemmata des aktuellen Artikels. In der Wortgeschichte zum Wort Bourgeois kann ein Verweis auf Kleinbürger darum als Gegensatz markiert werden:

<Verweis Typ="Gegensatz">
  <Verweistext/>
  <Verweisziel>Kleinbürger</Verweisziel>
</Verweis>

Solche Auszeichnungen eröffnen die Möglichkeit, korrespondierende Semantiken für Verweise auf andere Artikel zu berechnen. Das Gros der Hinweise diesen Typs speist sich aus einer solchen Berechnung, die einer ganz einfachen Logik folgt: Wenn im Artikel Bourgeois ein Verweis auf das Wort Kleinbürger als Gegensatz markiert wurde, muss im Artikel Kleinbürger ein Verweis auf Bourgeois ebenfalls einen Gegensatz markieren. (Asymmetrische Relationen werden bei dieser Berechnung natürlich entsprechend behandelt. Wird im Artikel Schiff das Wort Segelschiff als Hyponym markiert, wird für den Artikel Segelschiff vorgeschlagen, Schiff als Hyperonym zu kennzeichnen.)

Da die Semantiken bei WGd nicht mit den Bedeutungen, sondern mit den Lemmata verknüpft sind, kann es Grenzfälle geben, auf die diese Umkehrlogik nicht zutrifft. Darum müssen die Vorschläge von den Autor*innen manuell gesichtet und umgesetzt werden.

Die besondere Form des Paarartikels und der Umstand, dass die Semantiken nicht mit einzelnen Bedeutungen, sondern mit den Lemmata verknüpft sind, kann zu dem seltenen Fall führen, dass es sinnvoll ist, zwei verschiedene, einander normalerweise ausschließende Semantiken an ein und denselben Verweis zu hängen. So wird das Wort Kleinbürger zusammen mit dem Wort Großbürger in ein und demselben Artikel behandelt. Da sich Bourgeois, das in einem eigenen Artikel behandelt wird, in unterschiedlicher Weise auf diese beiden Wörter bezieht, kann ein Link aus dem Artikel Großbürger/Kleinbürger auf Bourgeois sowohl als Gegensatz als auch als Synonym gekennzeichnet werden, denn in Bezug auf Großbürger ist Bourgeois synonym, in Bezug auf Kleinbürger aber ein Gegensatz. Darum wird mitunter vorgeschlagen, mehr als eine Semantik an ein und denselben Verweis zu hängen:

<Verweis Typ="Gegensatz Synonym">
  <Verweistext/>
  <Verweisziel>Bourgeois</Verweisziel>
</Verweis>

Daneben gibt es Vorschläge, die sich aus Verweisen desselben Artikels speisen. Wurde im Artikel zum Wort Jetset einmal ein Verweis auf Upper Ten als Gegensatz markiert, sollten alle Verweise dieses Artikels, die auf Upper Ten zeigen, das Attribut @Typ="Gegensatz" haben. Ist dem nicht der Fall, wird darauf hingewiesen.

Die Auszeichnung lexikalischer Semantiken ist wichtig für die Darstellung der Verweiscluster im Artikel. In diesen werden nur diejenigen lexikalischen Semantiken abgebildet, die im XML-Code des jeweiligen Artikels wirklich getaggt wurden.

Allgemeine Fehler

Dateiname korrigieren

Für die Namen der XML-Dateien gibt es feste Regeln, die vom Artikeltyp und den behandelten Hauptlemmata abhängen. Folgt ein Dateiname nicht den Regeln, wird ein Hinweis diesen Typs ausgegeben. Der korrekte Dateiname wird vorgeschlagen und kann durch einfachen Klick kopiert werden.

Artikel-ID korrigieren

Tag:<Artikel>

Ebenso wie der Dateiname folgt auch die Vergabe der Artikel-ID festen Regeln. Wurde gegen diese Regeln verstoßen, wird ein Hinweis ausgegeben. Zwei Fälle können dabei auftreten:

ID formal falsch
In diesem Fall wird die korrekte ID vorgeschlagen, die man durch Klick kopieren und als Wert des @xml:id im Tag <Artikel> verwenden sollte.

ID bereits vergeben
IDs müssen zwingend einmalig sein. Darum darf ein und dieselbe ID auch nicht für mehrere Artikel genutzt werden. Passiert dies dennoch, muss eine der IDs zwingend geändert werden.

Die Vergabe korrekter Artikel-IDs ist von größter Wichtigkeit, da die Artikel von WGd automatisch in die Wortforschungsdatenbank des GBV eingespeist werden. Fehlermeldungen diesen Typs sollten darum umgehend und unbedingt korrigiert werden, denn es handelt sich in diesem Fall wirklich und wahrhaftig um Fehler.

Literaturtitel ergänzen

Skopus:<Literatur>

Tag:<Literaturtitel>

Der Literaturtitel wurde in der Datei share/Literaturliste.xml nicht gefunden. Wurde der Titel unter der genannten ID bereits in die Literaturdatenbank von Zettel’s Traum übertragen, kann dieser Fehler getrost ignoriert werden.

Beleg nicht zitiert

Skopus:<Belegreihe> bzw. <Wortgeschichte>, <Wortgeschichte_kompakt>

Attribut:@xml:id

Der im Hinweis genannte Beleg taucht zwar in der Belegreihe auf. In <Wortgeschichte> oder <Wortgeschichte_kompakt> findet sich jedoch kein <Belegreferenz>-Tag, dessen @Ziel auf den Beleg verweist.

Kommentare

XML-Kommentar

Skopus:<Artikel>

Knoten:<!-- -->

Die Kommentare werden aufgelistet, damit sie nicht in Vergessenheit geraten. Für den Fall, dass Sie nur Kommentare sehen wollen, in denen auch Autor*innentext steht, gibt es in der Filterleiste die Option

Ist die Option aktiviert, werden jene Kommentare ausgeblendet, in denen nur XML-Tags und Whitespace (Leerzeichen, Zeilenumbrüche, Tabs) zu finden sind.

Aufbau der Ansicht

Mithilfe der Clusterei können Sie die WGd-Verweiscluster verwalten. Die Ansicht besteht aus drei Sektionen, die über Icons in der Kopfleiste angesteuert werden:

  • In der Sektion Modulieren können Sie Clusterzentren modulieren, also ermitteln, welche Verlinkungen in welchen XML-Dateien ergänzt werden müssen, um ein gewünschtes Clusterzentrum zu erhalten.
  • In der Sektion Vergleichen werden einander ähnliche Cluster aufgeführt. Die XML-Dateien, die zu den Lemmata der Clusterzentren gehören, können von dort mit einem Klick zur Modulieren-Sektion hinzugefügt werden.
  • In der Sektion Überprüfen werden alle Cluster eines Themenfeldes so dargestellt wie in der WGd-Artikelübersicht auf zdl.org. Im Unterschied zur Online-Übersicht kann man hier jedoch leicht sehen, welche Cluster sich geändert haben und in welchen neue Lemmata hinzugekommen sind.

Sektion Modulieren

Diese Sektion ist auch ohne Filter voll nutzbar. Werden Filter ausgewählt, beziehen sich diese nur auf das Suchformular. Die zum Suchwort passenden Ergebnisse werden dann nach Autor*in und/oder Themenfeld gefiltert.

Das Grundprinzip eines jeden Clusterzentrums ist, dass die in ihm stehenden Hauptlemmata eine rekursive Beziehung zueinander haben: Jedes Lemma muss auf jedes andere Lemma im selben Zentrum verweisen. Nur wenn dies der Fall ist, bildet sich ein Clusterzentrum. Um den Saum und das Umfeld müssen Sie sich nicht kümmern; schauen Sie nur auf das Clusterzentrum!

Wenn Sie ein Zentrum modulieren wollen, gehen Sie wie folgt vor:

  1. Über das Suchfeld suchen Sie die Lemmata, die dem Zentrum angehören sollen.
  2. Klicken Sie auf die passenden Vorschläge in der Trefferliste.

  3. Unter der Vorschau des Zentrums sammeln sich nun nach und nach Artikelblöcke an. Denn das, was Sie zur Modulieren-Ansicht hinzufügen, sind nicht etwa Lemmata, sondern XML-Dateien. Alle in ihnen behandelten Hauptlemmata werden für die Modulierung herangezogen.

    (Nebenlemmata spielen für die Modulierung keine Rolle, da sie keine rekursiven Verweise erzeugen können. Man kann zwar auf Nebenlemmata verweisen, aber nicht von Nebenlemmata zu anderen Lemmata.)

    Oben im Artikelblock steht der Name der XML-Datei, daneben finden Sie drei Icons:

    Öffnet die XML-Datei im Vorschaufenster

    Öffnet die XML-Datei im Standard-XML-Editor Ihres Betriebssystems

    Entfernt den Artikelblock aus der Modulierung

    Auf die im Artikelblock alphabetisch aufgelisteten Lemmata wird in der Datei bereits verwiesen. Klicken Sie auf einen der blauen Links, öffnet sich eine Tabelle, in der Sie sehen, in welcher Zeile sich ein Verweis auf das genannte Lemma befindet.

  4. Im Kopf der Artikelblöcke werden sich nach und nach rote Links sammeln. Befinden sich in einem Artikelblock rote Links, wird auch der Dateiname dieses Blocks in einen roten Link verwandelt. Diese roten Links stehen für die Lemmata, die in der XML-Datei ergänzt werden müssen, um das Clusterzentrum zu erhalten, das Sie gerade modulieren.

    Durch Klick auf einen roten Link, wird der XML-Code des Verweises in die Zwischenablage kopiert. Sie können den Code dann direkt in die XML-Datei unterhalb von <Verweise Typ="Wortfeld"> einfügen. Klicken Sie auf den roten Link in der Überschrift, werden alle für den Artikel vorgeschlagenen Verweise en bloc kopiert.

  5. Noch bevor Sie die vorgeschlagenen Verweise in den XML-Dateien ergänzen, können Sie sich eine Vorschau berechnen lassen, um zu sehen, ob Ihnen das Ergebnis auch zusagt.

    Klicken Sie in der Kopfleiste auf das Icon und wählen Sie neue Berechnung starten aus. Die Option Modulierung mit einbeziehen wurde von QS bereits automatisch angestellt, wenn in der Sektion Modulieren neue Verweise vorgeschlagen werden, sprich: wenn dort irgendwo ein roter Link erscheint. Klicken Sie auf den Button Vorschau und die Cluster werden berechnet. Das Ergebnis können Sie sich dann in der Sektion Überprüfen ansehen.

Um eine Modulierung zurückzusetzen können Sie auf das Icon im Suchfeld klicken. Es entfernt alle Artikelblöcke aus der Modulierung.

Haben Sie sich eine Vorschau berechnen lassen, bleibt diese jedoch aktiv. Durch Klick auf das Icon in der Kopfleiste können Sie die Vorschau ausstellen. Danach arbeitet QS wieder mit den Clustern, die aktuell in der Artikelübersicht auf zdl.org angezeigt werden.

Clusterzentren können sinnvoll nur in eine Richtung moduliert werden: Sie können kleinere Zentren leicht zu größeren verschmelzen. Der umgekehrte Weg geht zwar theoretisch auch, dafür müssten Sie aber Verweise streichen. Und da es ja einmal einen Grund gab, aus dem heraus Sie diese Verweise gesetzt haben, sollten Sie diesen Weg normalerweise nicht beschreiten. Wollen Sie es dennoch versuchen, müssten Sie diejenigen rekursiven Verweisbeziehungen rückgängig machen, die dafür sorgen, dass ein Clusterzentrum entsteht. Dazu müssen Sie wissen, in welchen Artikeln auf welche Lemmata verwiesen wird. Die Sektion Modulieren kann Ihnen hierbei helfen: Die blauen Links in den Artikelblöcken öffnen eine kleine Tabelle, in der steht, in welchen Zeilen der XML-Datei auf das genannte Lemma verlinkt wird. So finden Sie die Verweise, die Sie streichen müssen. Das Ergebnis dieser mühsamen Arbeit wird aber nur sein, dass Sie viele, einander sehr ähnliche Cluster bekommen. Fangen Sie damit gar nicht erst an!

Sektion Vergleichen

Für diese Sektion muss zwingend ein Themfeld-Filter eingestellt werden. Zusätzlich kann auch der Autor*innen-Filter genutzt werden. Mit seiner Hilfe kann man sich nur diejenigen Cluster anzeigen lassen, in deren Zentrum mindestens ein Lemma der ausgewählten Autor*in zu finden ist.

In der Sektion werden einander sehr ähnlich Cluster aufgelistet und nebeneinandergestellt. Je ähnlicher sich die Cluster sind, umso eher bieten sie sich zum Verschmelzen an. Der angegebene Prozentwert bezieht sich auf das Verhältnis des kleineren zum größeren Cluster: 100 % bedeutet, dass 100 % der Lemmata des kleineren Clusters auch im größeren Cluster auftauchen, bei 75 % wären dies nur drei Viertel der Lemmata. Die Anzeige hat einen Schwellenwert von 75 %. Cluster, die im Vergleich eine geringere Ähnlichkeit haben, werden nicht gelistet.

Klicken Sie auf das Icon werden alle XML-Dateien, in denen sich die Hauptlemmata der nebeinander stehenden Cluster finden, zur Sektion Modulieren hinzugefügt. Die hinzugefügten Hauptlemmata werden dann blau hinterlegt. (Nebenlemmata spielen für die Modulierung keine Rolle, weswegen sie in dieser Ansicht ausgegraut sind.)

Das Icon hat sich nun geändert: . Klicken Sie es erneut an, wird der Vorgang umgekehrt – was nicht notwendigerweise gleichbedeutend mit rückgängig machen ist! Denn nun werden alle XML-Dateien, die zu den Lemmata in den nebeneinanderstehenden Clusterzentren gehören, aus der Sektion Modulieren entfernt.

Sektion Überprüfen

Für diese Sektion muss zwingend ein Themfeld-Filter eingestellt werden. Zusätzlich kann auch der Autor*innen-Filter genutzt werden. Mit seiner Hilfe kann man sich nur diejenigen Cluster anzeigen lassen, in deren Zentrum mindestens ein Lemma der ausgewählten Autor*in zu finden ist.

In der Sektion Überprüfen werden die Cluster analog zur Artikelübersicht auf zdl.org aufgelistet. Interessant ist diese Sektion v. a. im Zusammenhang mit der Funktion zur Berechnung der Vorschau. Haben Sie eine Vorschau berechnet, werden Cluster, die zuvor nicht existiert oder sich verändert haben, rot umrandet. Und Lemmata, die zuvor in keinem Cluster zu finden waren, werden sogar rot hinterlegt.

Mithilfe des Icons in der Kopfleiste können Sie durch die geänderten bzw. neuen Cluster springen. Dieses Icon ist nur sichtbar, wenn die Sektion Überprüfen aktiv ist.

Mitunter werden Sie auf rot umrandete Cluster stoßen, bei denen Sie keine visuelle Differenz zu einem bereits zuvor bestehenden Cluster erkennen können. Dies liegt daran, dass der Unterschied sehr subtiler Natur ist. In einem solchen Fall hat sich nämlich die Anzahl der Clusterpunkte geändert, mit deren Hilfe die Lemmata im Cluster gewichtet und sortiert werden. Die Änderung hatte dann aber einen so geringen Umfang, dass sie für die Visualisierung des Clusters keine Rolle spielte.

Sind Cluster ausgegraut werden diese in der Artikelübersicht auf zdl.org nicht angezeigt. Die Lemmata des ausgegrauten Clusterzentrums tauchen in solchen Fällen alle mindestens einmal in bereits weiter oben gedruckten Clustern auf. Cluster werden nur ausgegraut, wenn kein Autor*innen-Filter eingestellt ist, wenn also alle Cluster zu einem Themenfeld in der Anzeige sichtbar sind.

Vorschau berechnen

Normalerweise arbeitet die Clusterei mit den WGd-Verweisclustern, wie sie in der aktuellen Version der Datei Artikel.json zu finden sind. Dazu wird die Artikel.json aus dem Repository eingelesen. Die in ihr zu findenden Cluster werden dann für die Darstellung in der Clusterei genutzt.

Mithilfe des Icons in der Kopfleiste können Sie sich aber eine Cluster-Vorschau berechnen lassen, auf die dann in den Sektionen Vergleichen und Überprüfen zurückgegriffen wird. (Für die Sektion Modulieren spielen die bestehenden Cluster keine Rolle. In ihr geht es nur darum, was wirklich in den XML-Dateien steht.)

Bei der Berechnung werden die Lemmata aller XML-Dateien, von denen QS derzeit Kenntnis hat, mit einbezogen. Das heißt, dass auch Dateien berücksichtigt werden, die im Ordner ignore liegen.

Klicken Sie auf das Icon erscheint ein Popup-Fenster, in dem Sie die Berechnung durch ein paar Einstellungen beeinflussen können:

Modulierung mit einbeziehen

Ist diese Einstellung aktiv, tut QS so, als hätten Sie die rot eingefärbten Lemmata, die in der Sektion Modulieren vorgeschlagen werden, bereits in den XML-Dateien ergänzt. Für die Verweise wird dabei von einem Gewicht von 3 Punkten ausgegangen. Bei der Berechnung wird also simuliert, dass die Verweise bereits zu den Wortinformationen hinzugefügt wurden.

Cluster-Verweise ignorieren

Ist diese Einstellung aktiv, werden in allen XML-Dateien die Verweise mit dem Attribut @Typ="Cluster" bei der Berechnung der Vorschau ignoriert. Mit dieser Option können Sie also bereits vorgenommene Modulierungen temporär zurücksetzen. Ob das sinnvoll ist, steht auf einem anderen Blatt.

Die Berechnung der Cluster-Vorschau ist trotz diverser Optimierungen im Programmcode ziemlich rechenintensiv. Darum können Sie zwischen der Cluster-Vorschau und den im Repository zu findenden Clustern hin und her schalten, ohne jedes Mal eine Neuberechnung anstoßen zu müssen. Wählen Sie hierfür diese Option aus:

Allgemeines zur Suche

Standardmäßig werden alle XML-Dateien durchsucht, ganz gleich ob sie im Ordner articles oder ignore liegen. Die Suchwörter werden dabei einzeln behandelt, sprich anhand der Leerzeichen aufgetrennt. Anstoßen können Sie die Suche über den Suchbutton oder Enter im Suchfeld.

Die Suche ist auf 5000 Treffer beschränkt. Das klingt nach viel. Wenn Sie allerdings sehr unspezifische, kurze Suchwörter eingeben, ist diese Zahle rasch erreicht, da in den XML-Dateien jede Zeile, in der eines der Suchwörter zu finden ist, einen Treffer produziert, egal ob eines oder mehrere der Suchwörter auftauchen. Suchen Sie nach die das werden Sie darum niemals ein Ergebnis bekommen.

Normalerweise wird nur im Autor*innen-Text der XML-Dateien gesucht. Verwenden Sie in einem Suchausdruck aber die Zeichen < oder > werden auch die XML-Tags durchsucht. In der Trefferliste erscheint dann nicht der um Tags bereinigte Autor*innentext, sondern XML-Code.

Sie können den Umfang der Suche einschränken, indem Sie die Anzahl der XML-Dateien mithilfe der Filter einschränken, sich der Möglichkeiten der Suchsyntax bedienen und/oder Einstellungen aus der erweiterten Suche nutzen.

Erweiterte Suche

Die erweiterte Suche können Sie mit dem Icon rechts im Suchfeld öffnen (s. zu den anderen Icons den Abschnitt Kopfleiste). Die Einstellungen für die erweitere Suche öffnen sich außerdem automatisch, wenn Sie das globale Tastaturkürzel Strg + F benutzen. Sobald Sie eine Suche anstoßen, schließen sich die erweiterten Einstellungen.

Die vorgenommenen Einstellungen werden gespeichert und nach dem Neustart des Programms wiederhergestellt. Sind Einstellungen aktiv, die zu einer Verringerung der Trefferzahl führen, wird das Icon zum Öffnen der erweiterten Suche rot dargestellt: .

Die Einstellungen:

alle Suchwörter in derselben Zeile

Ist die Einstellung aktiviert, produzieren nur Zeilen in den XML-Dateien Treffer, in denen alle eingegebenen Suchwörter zu finden sind. Aktivieren Sie die Einstellung und suchen Sie nach die das, werden Sie (Stand Dezember 2022) zwar immer noch mehr als 3000 Treffer bekommen. Da dies die Maximalschwelle von 5000 Treffern klar unterschreitet, werden diese aber angezeigt.

Mit dieser Option können Sie die Suchtreffer sukzessive einschränken, denn für die nächste Suche werden nicht mehr alle XML-Dateien durchsucht, sondern nur noch diejenigen, die nach der aktuellen Suche einen Treffer produzierten. Welche Dateien dies sind, sehen Sie in der Ergebnisleiste. QS deaktiviert diese Einstellung nach der nächsten Suche automatisch.

Darüber hinaus können Sie den Suchbereich einstellen. Normalerweise werden die XML-Dateien von der ersten bis zur letzten Zeile durchsucht. Sie können die Suche aber auch auf bestimmte Abschnitte in den XML-Dateien einschränken. Mehrfachauswahl ist möglich.

  • Wortgeschichte: Durchsucht den Texte innerhalb von <Wortgeschichte> und <Wortgeschichte_kompakt>
  • Belege: Durchsucht den Texte innerhalb von <Belegreihe>
  • Bedeutungsgerüst: Durchsucht den Texte innerhalb von <Lesarten>
  • Wortinformationen: Durchsucht den Texte innerhalb von <Verweise>

Suchsyntax

QS geht davon aus, dass Sie Suchwörter, die durch Leerzeichen voneinander getrennt sind, einzeln behandeln möchten. Darum wird nach diesen Wörtern standardmäßig auch einzeln gesucht. Die Wörter werden beim Suchen durch ein logisches Und miteinander verknüpft.

Geben Sie also alternative Fakten ein, sucht QS nach alternative und Fakten. Nur wenn in einer XML-Datei beide Wörter einen Treffer produzieren, wird die Datei in den Suchtreffern gelistet.

Die Suche ist immer trunkiert. Gesucht wird also nicht nach ganzen Wörtern, sondern nach Buchstabenfolgen, die links und rechts erweitert sein können.

Geben Sie als Suchwörter alternativ fakten ein, findet das erste Wort auch die Formen alternative, alternativen, alternativer usw.

Taucht in einem Suchwort ein Großbuchstabe auf, wird für dieses Wort zwischen Groß- und Kleinschreibung unterschieden. Ansonsten werden Groß- und Kleinschreibung ignoriert.

Eine Suche nach fakt produziert darum mehr Treffer als eine nach Fakt. Im zweiten Fall werden nur Wörter wie Fakten, Faktum oder Faktor gefunden, wohingegen im ersten Fall zusätzlich auch faktuell, faktisch, aber auch kontrafaktisch gefunden werden.

Als Phrase können mehrere Wörter mithilfe von nicht-typographischen Anführungseichen gesucht werden.

Eine Suche nach "alternative Fakten" oder 'alternative Fakten' sucht die durch Leerzeichen voneinander getrennten Wörter dann nicht mehr einzeln, sondern als Phrase. Es werden nur Treffer ausgeworfen, in denen alternative Fakten steht. Aber auch in diesem Fall wird das Ergebnis links und rechts erweitert. Der Suchausdruck würde also auch auf alternative Faktenhuberei passen.

Suchen Sie im XML-Code ein Attribut mit einem bestimmten Wert, müssen Sie sich der einfachen Anführungszeichen bedienen, da die doppelten Anführungszeichen Teil des Suchtexts sind. Wenn Sie alle als englisch markierten Textteile finden wollen, suchen Sie also nach 'Typ="engl"'.

Spezialisten können auch reguläre Ausdrücke verwenden. Die Ausdrücke müssen mit / beginnen und enden. Hinter dem schließenden / darf zusätzlich noch der optionale i-Schalter stehen. Mit dem i-Schalter weist man den Ausdruck an, nicht auf Groß- oder Kleinschreibung zu achten.

Eine Suche mit einem regulären Ausdruck wie /alternativen? fakt(en)?/i ist immer eine Phrasensuche. In diesem Fall besteht aber eine gewisse Variabilität hinsichtlich der Schreibung von alternativen, das mit oder ohne n geschrieben sein darf. Außerdem trifft der Ausdruck auf fakt oder fakten zu. Durch den i-Schalter am Ende wird nicht auf Groß- oder Kleinschreibung geachtet. Aber auch hier gilt: Die Suche ist trunkiert.

Suchen Sie nach einem ganzen Wort, können Sie einen regulären Ausdruck mit sogenannten string boundaries verwenden. Eine Suche nach /\bFakt\b/ findet das Wort Fakt nur dann, wenn davor und dahinter z. B. Leer- oder Satzzeichen stehen. Aber Vorsicht! Die Eszett-Ligatur und die Umlaute gelten als Wortgrenze. (Die Definition von string boundaries ist auf niedrige ASCII-Zeichen beschränkt.) Eine Suche nach /\bMaß\b/ wird darum nicht das gewünschte Ergebnis bringen: Sie werden auch Maßnahme finden. Dies gilt für solche Suchen umso mehr, als QS automatisch gewisse Standardvarianten in die Suchausdrücke einbaut. Wird ß eingegeben, wird z. B. automatisch nach (ß|ss) (ß oder ss) gesucht.

Sie können reguläre Ausdrücke frei mit anderen Suchausdrücken kombinieren. Eine Suche nach Trump Lüge /alternativen? fakt(en)?/i stellt kein Problem dar und produziert sogar Treffer.

In der Trefferliste werden die Stellen, an denen das Suchwort gefunden wurde, normalerweise hervorgehoben. Verwenden Sie einen regulären Ausdruck und Suchen Sie im XML-Code kann dies aber mitunter scheitern.

Einstellungen: Daten

Zeitstrahldaten

Die Zeitstrahldaten spielen für Ihren Arbeitsalltag normalerweise keine Rolle. Sie können diese Einstellung also im Prinzip ignorieren. Benötigt werden die Zeistrahldaten allerdings für den Export der Artikel.json, um den sich die nicht spezialisierten Mitarbeiter*innen dann kümmern müssen, wenn die Techniker*in längerfristig ausfällt.

Bei der Artikel.json handelt es sich um eine für die Webseite unverzichtbare Datendatei, ohne die eine Publikation neuer Wortgeschichten nur eingeschränkt sinnvoll ist, weil viele Funktionen auf der Webseite von ihren Inhalten abhängen. Sie liegt im Repository unter resources/Artikel.json.

QS kann diese Datei zwar erzeugen, aber anders als ihre übrigen Inhalte können die Zeitstrahldaten nicht von der App berechnet werden. Wird die Datei ohne Zeitstrahldaten exportiert, kann auf der Webseite kein Zeitstrahl angezeigt werden.

Die Zeitstrahldaten liegen auf dem Netzlaufwerk. Unter Windows wäre das unter W:\Software\Entwicklungsumgebung WGd\wgd.local\scr\Zeitstrahl\data\data.json. Sie müssen einfach nur auf das Icon klicken und die Datei auswählen.

ZDL-Repository

Der Gebrauch der Publikations-Funktionen Export der Artikelübersicht und Export der Terminologie kann durch diese Einstellung erleichtert werden. Die Dateien, die durch die beiden Funktionen erzeugt werden, müssen nämlich im ZDL-Repository gespeichert werden. Wurde in dieser Einstellung zuvor der Pfad zum Repository hinterlegt, wird beim Export automatisch der korrekte Unterordner ausgewählt und man muss nur noch das Speichern (und Überschreiben) der Dateien bestätigen.

Einstellungsdaten

Mithilfe der Buttons Exportieren und Importieren können Sie – Sie ahnen es schon – die Programmeinstellungen ex- oder importieren.

Dies ist immer dann nützlich, wenn Sie abwechselnd an mehreren Rechnern arbeiten und Ihre Daten zwischen den Rechnern synchronisieren möchten. Denn die Einstellungen umfassen nicht nur die Konfiguration in der Filter- oder Kopfleiste. Zu ihnen gehören auch die in der Ansicht Hinweise gesetzten Markierungen. Vom Export ausgenommen sind die Git-Konfiguration und der Pfad zu den Zeitstrahldaten.

Markierungsdaten

Mit dem Button Löschen können Sie alle Markierungsdaten entfernen. Markierungen können Sie links neben den Hinweisen setzen, um sich so z. B. zu merken, dass Sie einen Hinweis bereits überprüft haben und nicht weiterverfolgen wollen. So markierte Hinweise lassen sich mit dem Abwählen der entsprechenden Filter ausblenden.

Die Markierungen gehören zu den Einstellungsdaten und können mit diesen ex- und importiert werden.

Analyse-Cache

Die Funktionen von QS beruhen alle auf Daten, die im Rahmen einer Analyse der im Repository liegenden XML-Dateien erhoben werden. Diese Analyse ist rechenintensiv, weswegen die App ihr Ergebnis in einem Cache speichert, um sie möglichst selten durchführen zu müssen. (Genauer gesagt gibt es für jeden Branch einen eigenen Cache, also z. B. einen für den Branch master und einen für preprint.)

Wurde der Cache einmal erstellt, wird die Analyse nur noch für neue oder geänderte XML-Dateien durchgeführt. Eine Ausnahme stellt allerdings die Berechnung der Hinweise dar. Diese werden jedes Mal komplett neu erstellt, wenn sich eine einzige XML-Datei geändert hat. (Dieses Verhalten wurde deswegen programmiert, weil die Hinweise sich auf dateiübergreifende Phänomene beziehen. Änderungen in einer Datei können Auswirkungen auf ein Vielzahl von Hinweisen in vielen verschiedenen Dateien haben.)

Komplett neu erstellt wird der Cache auch, wenn Sie eine neue Programmversion zum ersten Mal starten.

Klicken Sie auf den Button Löschen wird der Cache gelöscht und umgehend neu berechnet. Die Option ist eher für Entwickler gedacht und sollte für Ihren Arbeitsalltag nicht von Bedeutung sein.

HTML-Cache

Das Vorschaufenster ruft eine bestimmte Seite auf zdl.org auf, um die die Vorschauansicht eines Artikels zu erzeugen. Die empfangenen Daten werden in einem Cache gespeichert, um spätere Aufrufe zu beschleunigen. Wenn es ein größeres Update auf zdl.org gibt, kann es sein, dass die Ansicht im Vorschaufenster nicht mehr korrekt funktioniert. In diesem Fall sollten Sie den Cache leeren, damit das Fenster beim nächsten Öffnen alle Dateien neu herunterlädt.

Command-Line Interface (CLI)

QS verfügt über diverse Kommandozeilenschalter, wodurch die App im Prinzip wie ein Kommandozeilenprogramm nutzbar ist (wenn auch mit vergleichsweise geringer Performance, weil zunächst ein verstecktes App-Fenster gestartet werden muss). Mit den zur Verfügung stehenden Argumenten lassen sich alle Funktionen aufrufen, die sich im Menü Publikation finden.

Die folgenden Schalter können beliebig miteinander kombiniert werden. Auf diese Weise lassen sich mit einem einzigen Aufruf prinzipiell alle vier Funktionen in einem Zuge ausführen. Die einzelnen Optionen sollten im Befehl jedoch nicht mehrfach auftauchen. Sollte dies einmal der Fall sein, nutzt QS immer den am weitesten hinten stehenden Wert (was natürlich nur bei Optionen relevant ist, die einen Parameter verlangen).

--export-out=[Ordner]

Übergibt den Zielordner, in den hinein die gewünschte Datei exportiert werden soll. Wird eine der folgenden --export-Optionen genutzt, muss --export-out zwingend angegeben werden. Der übergebene Pfad muss außerdem zwingend ein Ordner, kann aber sowohl absolut als auch relativ sein.

--export-artikel-json

Die Datei Artikel.json soll berechnet und in dem via --export-out definiert Ausgabeordner abgelegt werden.

--export-overview

Die Artikelübersicht soll berechnet und in dem via --export-out definiert Ausgabeordner abgelegt werden. Der Name der Ausgabedatei lautet index.tt.

--export-terminology

Die Übersichtsseite mit dem terminologischen Kerninventar soll berechnet und in dem via --export-out definiert Ausgabeordner abgelegt werden. Das Format kann via --export-terminology-type gesteuert werden. Der Name der Ausgabedatei lautet terminologie.[tt | html].

--export-terminology-type=[tt | html]

Steuert das Ausgabeformat für die Übersichtsseite mit dem terminologischen Kerninventar. Wird diese Option nicht genutzt, wird standardmäßig das Ausgabeformat tt gewählt.

--no-new

Neue Datein werden bei der Berechnung nicht berücksichtigt. Hierbei handelt es sich um Dateien, die im Repository noch nicht committet wurden. Dies sind zum einen XML-Dateien, die im Ordner ignore liegen, und zum anderen untracked files, die im Ordner articles liegen. Diese Option ist für die Exportfunktionen --export-artikel-json, --export-overview und --export-terminology relevant.

--transform-svg=[SVG-Datei]

Die übergebene SVG-Datei wird so transformiert, dass sie in Stil und Farbe dem Schema auf zdl.org entspricht. Die Ausgabedatei hat denselben Namen wie das Original, allerdings mit einem vorangestellten ! . Sie wird im selben Ordner wie das Original abgelegt.

Publikation

Die Funktionen im Menü Publikation sollen es den Mitarbeiter*innen des Projekts ermöglichen, neue Wortgeschichten weitgehend selbständig auf zdl.org zu publizieren. Sie dienen alle dazu, bestimmte Dateien aufzufrischen oder zu transformieren. Dies sind Aufgaben, um die sich normalerweise eine Techniker*in kümmert. Damit es im Notfall alle übernehmen können, die Git in Grundzügen beherrschen, wurden diese Funktion nach QS portiert.

ZDL-Repository klonen

Für die Funktionen Artikelübersicht und Terminologie ist es sinnvoll, einen lokalen Klon des ZDL-Repositorys zu haben. Denn diese Funktionen erzeugen Dateien, die im ZDL-Repository committet und gepusht werden müssen.

Wenn Ihr System keine Symlinks unterstützt (dies gilt bes. für Windows), kann das Repository nicht korrekt geklont werden. Klone, in denen die Symlinks von Git automatisch durch Dateien ersetzt werden, können nicht zum Erstellen von Commits genutzt werden!

Das Klonen des ZDL-Repositorys funktioniert im Prinzip genauso wie das des Wortgeschichten-Repositorys. Man kann sich jedoch diverse Einrichtungsschritte sparen, weswegen es genügt, die folgenden Befehle in der Git Bash (unter Windows) bzw. im Terminal (unter macOS) der Reihe nach auszuführen. Der gefettete Text in den Befehlen muss entsprechend angepasst werden:

$ git clone https://git.zdl.org/zdl/zdl.org.git
$ cd zdl.org
$ git config user.name "Petra Pan"
$ git config user.email peter.pan@adwgoe.de
$ git config pull.rebase true
$ git config core.editor nano
$ git switch --track origin/wgd

Nur unter macOS und Linux:

$ git config credential.helper cache

Der Klon hat zwei Branches: master und wgd. Nach dem Einrichten ist der Branch wgd ausgecheckt. Da auf gar keinen Fall in master gearbeitet werden darf, muss der Branch vor dem Erstellen neuer Commits nicht gewechselt werden. Da der Branch allerdings von mehreren Mitarbeitern genutzt wird, sollten Sie vor dem Erstellen von Commits ein $ git pull nicht vergessen.

Funktion Artikel.json

Die Artikel.json ist eine Datendatei, die zahlreiche Funktionen auf der Webseite füttert, die ohne eine aktuelle Version dieser Datei nicht korrekt laufen.

Update-Prozedere

  1. Zeistrahldaten mit QS verknüpfen
  2. Nach Branch master wechseln
  3. Im Kopf des Hauptfenster auf pull klicken.

  4. Funktionen > Artikel.json > Exportieren

    Die App schlägt als Ziel bereits den richtigen Ordner im Repository vor. Die ältere Version der Artikel.json einfach ersetzen.

  5. Git Bash (unter Windows), Terminal (unter macOS) oder einen beliebigen Terminal-Emulator (unter Linux) im Repository-Ordner öffnen

  6. Commit erstellen und pushen:

    $ git add resources/Artikel.json
$ git commit -m "Artikeldaten-Update"
$ git push

Für Techniker*innen zur Evaluation: QS liefert ein Node-Script mit, mit dessen Hilfe man die Artikel.json in menschenlesbares JSON verwandeln kann. Die Datei ist im Original stark größenoptimiert, weswegen viele Werte nur Indizes sind, die auf ein Objekt namens values zeigen. Das Script erzeugt eine Datei mit Einrückungen, in der die Indizes durch die tatsächlichen Werte ersetzt wurden. Es liegt im Installationsordner von QS im Unterordner resources. Benutzung:

$ node hrArtikel.js /PFAD/ZUM/REPOSITORY/resources/Artikel.json

Die Output-Datei wird in dem Ordner erzeugt, in dem die Artikel.json liegt.

Der Installationsordner von QS ist abhängig vom Betriebssystem:

  • Windows: C:\Users\BENUTZERNAME\AppData\Local\Programs\QS\
  • Linux: /opt/QS/

Funktion Artikelübersicht

Mithilfe dieser Funktion kann die Artikelübersicht auf zdl.org aufgefrischt werden. Die Darstellung der Artikelübersicht speist sich zum einen aus der Artikel.json und zum anderen aus einem HTML-Template, in dem die Lemmata aufgelistet werden. Das Template wird mit dieser Funktion erstellt.

Soll die Übersicht aufgefrischt werden (weil z. B. eine neue Wortgeschichte erschienen ist), muss die Datei, die nach dem Drücken von Exportieren zum Speichern angeboten wird, am richtigen Ort im ZDL-Repository gespeichert werden, nämlich unter:

/PFAD/ZUM/ZDL-REPOSITORY/root/wb/wortgeschichten/index.tt

Doch zunächst sollten Sie im Kopf des Programmfenster auf pull klicken.

Danach können Sie die Template-Datei erzeugen und die bereits bestehende Datei überschreiben, um im Anschluss die Änderungen zu committen und zu pushen:

$ git add root/wb/wortgeschichten/index.tt
$ git commit -m "wgdListe: Update"
$ git push

Der Dialog, der sich nach dem Drücken auf Exportieren öffnet, wählt automatisch den richtigen Speicherort aus, wenn in den Einstellungen zuvor der Pfad zum ZDL-Repository hinterlegt wurde.

Funktion Terminologie

Mithilfe dieser Funktion kann das terminologische Kerninventar auf zdl.org aufgefrischt werden. Die Anzeige dieser Seite (und auch die der Erläuterungsfenster in den Artikeln) speist sich komplett aus der Terminologie-Datendatei, die im Wortgeschichten-Repository liegt:

/PFAD/ZUM/ZDL-REPOSITORY/root/wb/WGd/terminologie.tt

Das Ganze funktioniert ganz gleich, wie das Update der Artikelübersicht:

Funktion Wortverlaufskurven

Mit dieser Funktion können die Wortverlaufskurven des DWDS für die Publikation einer Wortgeschichte transformiert werden. Bei der Transformation wird Folgendes geändert:

  • Stil und Farbe der Kurven werden an das Schema von zdl.org angepasst;
  • aus den Legenden werden überflüssige Wörter gelöscht (z. B. Gesamt).

Das Ergebnis der Transformation wird in einer neuen Datei gespeichert, die im selben Ordner abgelegt wird, in dem die zuvor geladene Wortverlaufskurve liegt. Der Name der neuen Datei beginnt mit ! . Er sollte also ganz oben im Ordner zu finden sein.
